# Phase 10 — UI Design Contract

> Visual and interaction contract for DriftPage, QuickTasksPage, and VerificationPage.
> Generated by gsd-ui-researcher. Consumed by gsd-ui-checker, gsd-planner, gsd-executor.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none — custom zinc dark theme |
| Preset | not applicable |
| Component library | none — hand-rolled Tailwind utility components |
| Icon library | none — unicode glyphs for expand/collapse (› / ⌄) |
| Font | Segoe UI, system-ui, sans-serif (set in index.css) |

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, badge inner padding (px-1.5 py-0.5) |
| sm | 8px | Row vertical padding (py-2), compact button padding (px-2 py-1.5) |
| md | 16px | Card padding (p-4), standard horizontal padding (px-4) |
| lg | 24px | Page container padding (p-6) |
| xl | 32px | — |
| 2xl | 48px | — (p-8 used only for stub placeholders — do not use in real pages) |
| 3xl | 64px | — |

Exceptions:
- Row min-height: 28px (`min-h-7`) for tree/list items — inherited from DocsPage pattern
- Left accent border on phase rows: 3px (`border-l-[3px]`) — established in DashboardPage
- Border radius on cards/badges: `rounded-md` (6px), `rounded` (4px) for inline badges

Source: `DashboardPage.tsx` p-6 container, p-4 cards, `DocsPage.tsx` min-h-7 rows, px-2 py-0.5 badges.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body / row text | 14px (text-sm) | 400 (normal) | 1.5 |
| Label / secondary | 12px (text-xs) | 400 (normal) | 1.4 |
| Section heading | 12px (text-xs) | 500 (medium), uppercase, tracking-wider | 1.4 |
| Card metric / stat | 24px (text-2xl) | 500 (medium) | 1.2 |

Notes:
- Phase titles use `font-mono font-medium text-[#cccccc]` (established in DashboardPage row pattern)
- Section labels use `font-medium uppercase tracking-wider text-[#858585]` (established in DocsPage)
- No heading larger than text-2xl appears on data pages — these are data tables, not marketing pages
- Do NOT introduce text-lg, text-xl, or text-3xl on these pages
- The text-2xl size provides sufficient visual differentiation for stat values without requiring a heavier weight

Source: `DashboardPage.tsx` phase rows, stat cards; `DocsPage.tsx` section headers.

Source: `DashboardPage.tsx` badge and border patterns, `DocsPage.tsx` selected item colors.

## Page-by-Page Interaction Contracts

### DriftPage (DRFT-02 through DRFT-05)

**Layout:** Full-width scrollable list inside `p-6` container. No sidebar. No drawer.

**Data source:** `activeProject.milestones.flatMap(m => m.phases)` from context — no extra fetch needed.

**Sort order:** MAJOR phases first, then MINOR, then NONE, then DEFERRED (DRFT-02).

**Row structure (DRFT-03):**
- Phase number: `font-mono font-medium text-[#cccccc]` — zero-padded to 2 digits
- Phase title: `font-mono font-medium text-[#cccccc]`
- Status label: `text-xs text-[#858585]` — same `statusLabel()` function as DashboardPage
- Drift badge: `rounded px-1.5 py-0.5 text-xs font-medium` with semantic colors above
- Plan age: `text-xs text-[#858585]` — "N days" or "—" when `plan_write_time` absent
- Last updated: `text-xs text-[#858585]` — formatted with `fmtDate()` helper

**Collapsed section (DRFT-05):**
- Phases with `status === "not_started"` AND `drift === "deferred"` are hidden by default
- Toggle button: `text-xs text-[#858585] hover:text-[#cccccc]` with `›` / `⌄` glyph — "Show N un-started" / "Hide un-started"
- Revealed rows rendered with `opacity-60` to visually distinguish from active phases

**No-project state:** `<div className="p-6 text-[#858585] text-sm">Add scan roots in Settings and select a project.</div>`

**Empty (all phases deferred):** Show toggle immediately — "Show N un-started phases" with count.

**Note on `plan_write_time`:** This field is not currently in `PhasePayload` in `api.ts`. The executor must either add it to the interface or compute plan age from `last_updated` as a fallback. This is a data-layer decision deferred to the planner.

### QuickTasksPage (QTSK-01 through QTSK-05)

**Layout:** Full-width scrollable list inside `p-6` container. No sidebar. No drawer.

**Data source:** `fetchQuickTasks(activeSegment.planningPath)` from `api.ts` — already implemented. Fetched on mount and on `activeSegment` change.

**Sort order:** `last_updated` descending — most recently active first (QTSK-05).

**Row structure (QTSK-02):**
- Title: `text-sm font-medium text-[#cccccc]`
- Status badge: `rounded px-1.5 py-0.5 text-xs font-medium` with semantic colors above (QTSK-03)
- Created date: `text-xs text-[#858585]` — formatted with `fmtDate()`
- Last updated: `text-xs text-[#858585]` — formatted with `fmtDate()`

**Row container:** `rounded-md border border-[#474747] bg-[#1e1e1e] p-4` — matches DashboardPage card style. Rows are NOT clickable in this phase (QTSK-06 is deferred).

**Loading state:** `<p className="text-[#858585] text-sm">Loading…</p>` inside `p-6` wrapper.

**Empty state (QTSK-04):**
- Heading: "No quick tasks yet"
- Body: "Quick tasks appear here when you run /gsd:quick in this project."
- Styling: `text-center p-8 text-[#858585] text-sm`

**Error state:** `<p className="text-red-400 text-sm">Could not load quick tasks. Check your project path and try refreshing.</p>`

**No-project state:** `<div className="p-6 text-[#858585] text-sm">Add scan roots in Settings and select a project.</div>`

### VerificationPage (VERIF-01 through VERIF-04)

**Layout:** Full-width scrollable list inside `p-6` container. No sidebar.

**Data source:** `activeProject.milestones.flatMap(m => m.phases)` from context — no extra fetch needed.

**Sort order:** By phase number ascending (verification is naturally sequential; matches roadmap reading order).

**Row structure (VERIF-02):**
- Phase number: `font-mono font-medium text-[#cccccc]` — zero-padded to 2 digits
- Phase title: `font-mono font-medium text-[#cccccc]`
- `has_validation` badge: "Validated" / "None" with semantic colors
- `nyquist_compliant` badge: "Pass" / "Fail" / "Unknown" with semantic colors
- `has_uat` badge: "UAT" / "No UAT" with semantic colors

**Expandable row (VERIF-03):**
- Clicking a row with `has_validation=true` toggles an inline expansion below the row
- Expanded area renders `validation_content` via `<ReactMarkdown remarkPlugins={[remarkGfm]}>` inside `prose prose-invert prose-sm max-w-none` — matches DashboardPage drawer pattern
- Toggle indicator: `›` (collapsed) / `⌄` (expanded) in `text-xs text-[#858585]` placed left of phase number
- Row with `has_validation=false` is NOT clickable (no expand behavior); cursor remains default

**Collapsed section (VERIF-04):**
- Phases with `has_validation=false` are separated into a secondary section, hidden by default
- Toggle button: `text-xs text-[#858585] hover:text-[#cccccc]` — "Show N phases without validation" / "Hide"
- Revealed rows rendered with `opacity-60`

**No-project state:** `<div className="p-6 text-[#858585] text-sm">Add scan roots in Settings and select a project.</div>`

**Empty (no phases):** `<div className="p-6 text-[#858585] text-sm">No phases found for this project.</div>`

## Shared Structural Patterns

### Page Container

All three pages use:
```
<div className="p-6">
  {/* page content */}
</div>
```

Matches DashboardPage. The `main` in ShellLayout already provides `overflow-auto`.

### Table/List Row Pattern

Phase rows use the established DashboardPage button-row pattern:
```
<button type="button" className="w-full rounded-md border border-[#474747] border-l-[3px] {statusBorderClass} bg-[#1e1e1e] p-4 text-left transition-colors hover:bg-[#2a2d2e]">
  <div className="flex items-center justify-between gap-2">
    {/* left: phase number + title */}
    {/* right: badges + dates */}
  </div>
</button>
```

For non-clickable rows (QuickTasks, Verification rows without validation), use `<div>` not `<button>`.

### Badge Pattern

All status/indicator badges follow:
```
<span className="rounded px-1.5 py-0.5 text-xs font-medium {colorClasses}">
  {label}
</span>
```

### Section Divider / Toggle Pattern

For collapsible sections (DRFT-05, VERIF-04):
```
<button type="button" className="mt-4 flex items-center gap-1 text-xs text-[#858585] hover:text-[#cccccc]">
  <span>{isOpen ? "⌄" : "›"}</span>
  <span>{isOpen ? "Hide un-started" : "Show N un-started"}</span>
</button>
```

## Accessibility

- All clickable rows use `<button type="button">` — not `<div onClick>` or `<tr onClick>`
- Non-clickable rows use `<div>` — not `<button>` with disabled state
- Expandable rows include `aria-expanded={isOpen}` on the toggle trigger
- Toggle sections include `aria-controls` pointing to the collapsible container id
- Badges are `<span>` — purely visual, no interactive role needed
- Color is never the sole indicator: drift MAJOR also shows the label "Major"; validated phases also show checkmark badge text

## Open Implementation Notes for Planner

1. **`plan_write_time` field:** `PhasePayload` in `api.ts` does not currently expose `plan_write_time`. DriftPage needs plan age in days. The planner must decide: add `plan_write_time?: string | null` to `PhasePayload` and update the API serializer, OR accept that plan age is omitted from the UI and DRFT-03 shows "—" for plan age. This is a backend/API typing decision, not a UI design decision.

2. **`plan_write_time` computation on backend:** Already implemented in Phase 9 (`_compute_drift`). The field just needs to be serialized into the API response if the planner chooses option A above.

3. **Shared helpers:** `fmtDate()` and `statusLabel()` are currently defined inside `DashboardPage.tsx`. The executor should extract them to a shared `src/utils.ts` module before implementing the new pages, to avoid duplication.

4. **No new routes needed:** All three pages already exist as stubs and are already registered in `ShellLayout.tsx` nav. The executor replaces stub content only.
