  <action>
**Step 1: Clean old dist**
Remove the stale `frontend/dist/` directory:
```
rm -rf frontend/dist
```

**Step 2: Run the build**
```
cd frontend && npm run build
```
This executes `tsc -b && vite build` per package.json scripts. The `tsc -b` step type-checks (should pass from Plan 01). The `vite build` step bundles into `frontend/dist/`.

If `tsc -b` fails, read the error output carefully. Common issues:
- If `@tailwindcss/typography` was not installed in Plan 01 and `prose` classes cause issues, that is a CSS issue not a tsc issue — tsc should still pass.
- If unused variable/parameter errors appear, they must be fixed in the offending file (tsconfig has `noUnusedLocals` and `noUnusedParameters` enabled).

**Step 3: Verify bundle exists**
Confirm `frontend/dist/index.html` exists and `frontend/dist/assets/` contains `.js` and `.css` files.

**Step 4: Verify bundle content**
Search the built JS bundle(s) for key strings that prove Phase 02-05 features are included:

```bash
# DocsPage component (Phase 03)
grep -l "DocsPage\|file-tree\|planning" frontend/dist/assets/*.js

# AppProvider / context (Phase 02+)
grep -l "AppProvider\|useApp\|AppCtx" frontend/dist/assets/*.js

# stateCurrentPosition (Phase 04/05)
grep -l "stateCurrentPosition\|state_current_position" frontend/dist/assets/*.js

# ReactMarkdown (Phase 03)
grep -l "ReactMarkdown\|react-markdown\|remarkGfm" frontend/dist/assets/*.js

# Drawer component (Phase 02 dependency)
grep -l "Drawer\|drawer" frontend/dist/assets/*.js

# SettingsPage (PERF-04)
grep -l "SettingsPage\|settings" frontend/dist/assets/*.js
```

Each grep should match at least one JS file in the bundle. If any grep returns no matches, investigate — the feature may have been tree-shaken (unlikely for route-level components) or the source file may have had issues.

**Step 5: Verify dist/index.html structure**
The built `frontend/dist/index.html` should contain a `<script>` tag referencing an asset in `assets/` and a `<link>` tag for CSS.
  </action>
