# Milestone v5.0: Installation and Distribution

**Status:** ✅ SHIPPED 2026-07-18
**Phases:** 17–18
**Total Plans:** 3

## Overview

v5.0 delivers a complete installation and upgrade story for GSD Monitor. A developer can install from scratch with a single PowerShell script, upgrade an existing installation with a standalone script, and see the app's version number in the sidebar footer at all times.

## Phases

### Phase 17: Installation & Upgrade Scripts

**Goal:** A developer can install GSD Monitor from scratch with a single PowerShell script, and upgrade an existing installation with a standalone upgrade script
**Depends on:** Phase 16
**Requirements:** INST-01, INST-02, INST-03, INST-04, INST-05, UPG-01, UPG-02, UPG-03

Plans:

- [x] 17-01-PLAN.md — `install.ps1`: prereq checks (Git, Python, Node, WebView2), clone, venv, pip install -e ., frontend build, .bat launcher, desktop shortcut, idempotent re-run; add `pyproject.toml` with hatchling + dynamic version
- [x] 17-02-PLAN.md — `upgrade.ps1` / `upgrade.bat`: valid-install guard, git pull, pip install -e ., npm ci && npm run build, already-up-to-date detection, success/error messaging

**Key decisions:**
- `pyproject.toml` uses `dynamic = ["version"]` read from `gsd_monitor/__init__.py` via hatch
- install.ps1 re-run detects `.git` presence and offers upgrade path (interactive Y/n) rather than re-cloning
- upgrade.ps1 uses `git pull` (not `git reset --hard`) — gentler than the start.bat auto-updater
- upgrade.ps1 is non-interactive by design (no `Read-Host`)

## Milestone Summary

**Key Decisions:**

- `pyproject.toml` with hatch dynamic version as the canonical version source — used by both pip install and the `/api/version` endpoint
- install.ps1 detects existing installation by `.git` presence and routes to upgrade branch rather than re-clone
- upgrade.ps1 separated from install.ps1 — standalone script users can run directly without re-running the full installer
- upgrade.ps1 uses git hash comparison for already-up-to-date detection (no network required after fetch)

**Issues Resolved:**

- No prior installation story — developers had to manually clone + set up venv + build frontend
- Version number was invisible — no way to know which build was running

**Issues Deferred:**

- Chocolatey / winget package distribution (Future Requirements in REQUIREMENTS.md)
- Auto-update check against GitHub releases (Future Requirements)
- Uninstall script (Future Requirements)
- Phase 18 has no planning directory — executed inline; ROADMAP + commit are the only records
- Phase 17 missing VERIFICATION.md — self-checks in SUMMARY passed but no formal verification artifact

**Technical Debt Incurred:**

- install.ps1 re-run path uses `Read-Host` (interactive) — INST-04 is met but not fully automatable
- install.ps1 upgrade branch uses `git reset --hard origin/master` (same footprint as the old start.bat auto-updater) — acceptable for installer context but heavier than upgrade.ps1's `git pull`
