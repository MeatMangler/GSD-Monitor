# Quick Task 260405-759: Fix stale data in the Projects/Docs UI

## Plan

### Task 1: DocsPage — refetch tree and content on project data refresh
**Files:** `frontend/src/pages/DocsPage.tsx`
**Action:**

The root cause: DocsPage's two fetch effects (tree fetch at line 69, file content fetch at line 85) are keyed only on `activeSegment?.planningPath` (and `selectedPath` for content). When the WebSocket fires `projects_updated`, context.tsx calls `reload()` which updates `groups` state, which recomputes `activeSegment` — but if `planningPath` string is unchanged (same project, same segment), the effects do not re-run and stale tree/content persists.

Fix by adding a monotonic refresh counter to serve as a cache-busting dependency:

1. Import `useRef` (already importing from "react").
2. Extract `groups` from `useApp()` (add it alongside `activeSegment, activeProject, loading`). The `groups` array reference changes on every `reload()` call in context.tsx since `fetchGroups()` always returns a new array.
3. Add a `refreshKey` state: `const [refreshKey, setRefreshKey] = useState(0)`.
4. Add a new `useEffect` that increments `refreshKey` whenever `groups` changes (but NOT on mount — use a ref to skip the initial render):
   ```typescript
   const isInitialMount = useRef(true);
   useEffect(() => {
     if (isInitialMount.current) {
       isInitialMount.current = false;
       return;
     }
     setRefreshKey((k) => k + 1);
   }, [groups]);
   ```
5. Add `refreshKey` to the dependency arrays of both fetch effects:
   - Tree fetch effect (line 69): `[activeSegment?.planningPath, refreshKey]`
   - File content fetch effect (line 85): `[activeSegment?.planningPath, selectedPath, refreshKey]`

This ensures that when `projects_updated` fires -> context reloads groups -> groups reference changes -> refreshKey increments -> both DocsPage effects re-run, fetching fresh tree and content from the API (which always reads from disk).

Do NOT change the reset effect (line 60, keyed on `planningPath`) — that correctly resets selection state only when switching projects.

**Verify:** `cd frontend && npx tsc --noEmit` exits 0. Then manual: start the app, open Docs page, create/delete a file under the active project's `.planning/` directory, confirm the sidebar tree updates within a few seconds without switching projects.
**Done:** DocsPage tree and content automatically refresh when filesystem changes are detected, without requiring a project switch.

### Task 2: Backend — re-scan on _dirty after refresh()
**Files:** `gsd_monitor/api/app.py`
**Action:**

In `RuntimeState.refresh()` (line 83-87), the method runs `discover_groups` once under the lock but does not check `_dirty` afterward. If `_on_fs_change` fires while `refresh()` holds `_refresh_lock`, the callback sets `_dirty = True` and returns (non-blocking acquire fails). But `refresh()` never re-checks `_dirty`, so that change is lost until another FS event arrives.

Fix by adding the same dirty-loop pattern that `_on_fs_change` already uses:

Replace the current `refresh()` method body:
```python
def refresh(self) -> None:
    with self._refresh_lock:
        while True:
            self._dirty = False
            s = self.settings_service.load()
            self.groups = self.discovery.discover_groups(s.scan_roots)
            if not self._dirty:
                break
        self.watcher.set_roots(list(s.scan_roots))
```

Key points:
- `self._dirty = False` at the top of the loop (same as `_on_fs_change`)
- After `discover_groups`, check `_dirty` — if True, loop again
- `set_roots` stays outside the loop (only needs to run once, roots do not change mid-refresh)
- The `s` variable from the last iteration is used for `set_roots` — this is fine since settings do not change within the lock

**Verify:** `cd D:/gsd-monitor-py && python -m pytest tests/ -x` passes. Review the method manually: confirm the while/dirty pattern matches `_on_fs_change`.
**Done:** `refresh()` re-scans if filesystem events arrived during its scan, eliminating the edge case where a single `_dirty` flag is silently dropped.
